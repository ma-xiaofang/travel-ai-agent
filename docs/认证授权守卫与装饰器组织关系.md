# 认证授权：守卫与装饰器组织关系

本文梳理 `apps/server/src/auth` 模块中**守卫（Guards）**与**装饰器（Decorators）**的协作关系，是后端访问控制（认证 + RBAC 授权）的核心设计说明。

## 一、整体设计思想：元数据标记 + 守卫读取（解耦）

两者通过**元数据 Key 常量**解耦协作：装饰器只负责「打标记」，守卫负责「读标记并执行校验」。彼此不直接引用，全靠 `Reflector` + 共享的 Key 常量串联。

```
装饰器（声明意图，写元数据）        守卫（读元数据，执行校验）
─────────────────────────         ─────────────────────────
@Public()  ──IS_PUBLIC_KEY──►      JwtAuthGuard（认证）
@Roles()   ──ROLES_KEY──────►      RolesGuard（授权）
@CurrentUser() ◄──req.user── 由 JwtStrategy.validate() 注入
```

## 二、成员清单与职责

### 装饰器（`decorators/` + 参数装饰器）

| 装饰器 | 类型 | 作用 | 关联 Key |
|--------|------|------|----------|
| `@Public()` | 方法/类元数据 | 标记路由免认证（登录、注册、健康检查等） | `IS_PUBLIC_KEY` |
| `@Roles('ADMIN')` | 方法/类元数据 | 声明路由所需角色 | `ROLES_KEY` |
| `@CurrentUser()` | 参数装饰器 | 注入 `req.user` 到方法参数 | —（直接读 request） |

### 守卫

| 守卫 | 注册方式 | 作用 | 读取的 Key |
|------|----------|------|-----------|
| `JwtAuthGuard` | **全局**（`app.module.ts` 的 `APP_GUARD`） | 校验 JWT、注入用户 | `IS_PUBLIC_KEY` |
| `RolesGuard` | **局部**（控制器 `@UseGuards`） | 校验角色 | `ROLES_KEY` |

### 支撑件

- **`JwtStrategy`**：Passport 策略，`validate()` 把 token 负载转成 `AuthUser` → 挂到 `req.user`
- **`AuthUser`**：`req.user` 的类型契约，被 `JwtStrategy`、`@CurrentUser`、`RolesGuard` 共同依赖
- **`AuthModule`**：`providers` 注册并 `exports` 三者（`JwtStrategy / JwtAuthGuard / RolesGuard`）供全局复用

## 三、运行时执行链路（一次受保护请求）

```
请求
  │
  ▼
① JwtAuthGuard（全局，最先执行）
   ├─ Reflector 读 @Public 元数据
   │    └─ 是公开路由 → 直接放行（跳过下面）
   └─ 否则 → 委托 Passport 'jwt' 策略
            └─ JwtStrategy.validate() → 返回 AuthUser → 注入 req.user
  │
  ▼
② RolesGuard（控制器级，认证之后）
   ├─ Reflector 读 @Roles 元数据
   │    └─ 未声明角色 → 放行
   └─ 否则 → 校验 req.user.role 是否在允许列表
            └─ 不匹配 → 抛 ForbiddenException('权限不足')
  │
  ▼
③ 控制器方法
   └─ @CurrentUser() 取出 req.user 注入参数
```

> 关键顺序：**先认证后授权**。`JwtAuthGuard` 全局先跑并注入 `req.user`，`RolesGuard` 才能在其基础上读取角色——这也是 `RolesGuard` 必须排在认证守卫之后的原因。

## 四、典型用法示例

```ts
// 公开接口：跳过 JWT 认证
@Public()
@Post('login')
login(@Body() dto: LoginDto) { /* ... */ }

// 仅 ADMIN 可访问：控制器级挂载 RolesGuard + 声明角色
@Controller('api/admin/users')
@UseGuards(RolesGuard)
@Roles('ADMIN')
export class AdminUsersController { /* ... */ }

// 业务接口：默认已被全局 JwtAuthGuard 保护，直接取当前用户
@Get('profile')
getProfile(@CurrentUser() user: AuthUser) {
  return user;
}
```

## 五、组织关系总结

1. **职责分层**：装饰器 = 声明式「标记」（What），守卫 = 命令式「执行」（How），两者关注点分离。
2. **解耦媒介**：靠 `IS_PUBLIC_KEY` / `ROLES_KEY` 字符串常量 + `Reflector.getAllAndOverride()`（方法级覆盖类级）连接，互不硬依赖。
3. **数据契约**：`AuthUser` 是贯穿「策略注入 → 守卫读取 → 装饰器取用」的统一类型。
4. **注册粒度**：认证守卫**全局兜底**（默认所有路由都要登录，`@Public` 例外）；授权守卫**按需局部**挂载（默认不限角色，`@Roles` 才生效）——典型的「白名单放行 + 黑名单限权」组合。

这是 NestJS 推荐的标准 RBAC 组织范式：`Strategy`（怎么验 token）→ `Guard`（要不要拦）→ `Decorator`（怎么声明规则 / 取数据），层次清晰、可组合、易扩展。
